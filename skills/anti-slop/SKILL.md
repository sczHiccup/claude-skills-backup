# Anti-Slop Skill

检测并消除自然语言、代码和设计中的通用AI生成模式（“slop”）。

## 什么是AI Slop？

AI slop指的是信号低质量、通用AI生成内容的特征模式：
- **文本**：过度使用的短语，如“深入探讨”，过多的流行词，元评论
- **代码**：通用变量名，明显的注释，不必要的抽象
- **设计**：千篇一律的布局，通用渐变，过度使用的视觉模式

该技能帮助识别并移除这些模式，以创建真实、高质量的内容。

## 何时使用此技能

在以下情况下应用反slop技术：
- 在交付前审查AI生成的内容
- 创建原创内容并希望避免通用模式
- 清理感觉通用的现有内容
- 为项目建立质量标准
- 用户明确请求slop检测或清理
- 内容有明显的通用AI生成迹象

## 核心工作流程

### 1. 检测Slop

**对于文本文件：**
```bash
python scripts/detect_slop.py <file> [--verbose]
```

这将分析文本并提供：
- Slop评分（0-100，分数越高越糟）
- 具体模式发现
- 可操作的建议

**手动检测：**
阅读适当的参考文件以获取详细模式：
- `references/text-patterns.md` - 自然语言slop模式
- `references/code-patterns.md` - 编程slop模式  
- `references/design-patterns.md` - 视觉/UX设计slop模式

### 2. 清理Slop

**自动清理（仅限文本）：**
```bash
# 预览更改
python scripts/clean_slop.py <file>

# 应用更改（创建备份）
python scripts/clean_slop.py <file> --save

# 激进模式（可能稍微改变含义）
python scripts/clean_slop.py <file> --save --aggressive
```

**手动清理：**
根据检测到的模式应用参考文件中的策略。

## 文本Slop检测与清理

### 高优先级目标

**立即移除：**
- "delve into" → 删除或替换为 "examine"
- "navigate the complexities" → "handle" 或删除
- "in today's fast-paced world" → 删除
- "it's important to note that" → 删除
- 关于文档本身的元评论

**简化冗长短语：**
- "in order to" → "to"
- "due to the fact that" → "because"
- "has the ability to" → "can"

**替换流行词：**
- "leverage" → "use"
- "synergistic" → "cooperative"
- "paradigm shift" → "major change"

### 质量原则

**直接：**
- 跳过前言和元评论
- 直接切入实际要点
- 剔除不增加意义的过渡词

**具体：**
- 用具体示例替换通用术语
- 指明具体事物，而不是“项目”、“东西”、“数据”
- 使用精确动词而不是模糊的行动词

**真实：**
- 变化句子结构和长度
- 主要使用主动语态
- 以适合上下文的语气写作，而不是企业通用的语气

## 代码Slop检测与清理

### 高优先级目标

**重命名通用变量：**
- `data` → 命名它代表的数据
- `result` → 命名结果包含的内容
- `temp` → 命名你临时存储的内容
- `item` → 命名是什么类型的项目

**移除明显的注释：**
```python
# Bad
# Create a user
user = User()

# Better - let code speak
user = User()
```

**简化过度设计的代码：**
- 移除不必要的抽象层
- 替换无目的使用的设计模式
- 简化简单任务的复杂实现

**改善函数名称：**
- `handleData()` → 你在处理数据时做了什么？
- `processItems()` → 具体处理了什么？
- `manageUsers()` → 进行了什么管理操作？

### 质量原则

**清晰胜于聪明：**
- 编写易于理解的代码
- 仅在性能分析显示需要时进行优化
- 优先选择简单解决方案而非复杂方案

**有意义的名称：**
- 变量名称应描述内容
- 函数名称应描述动作 + 对象
- 类名称应描述责任

**适当的文档：**
- 文档应说明原因，而不是内容
- 跳过自明代码的文档
- 将文档重点放在公共API和复杂逻辑上

## 设计Slop检测与清理

### 高优先级目标

**视觉slop：**
- 通用渐变背景（紫色/粉色/青色）
- 过度使用玻璃化或新拟态
- 无目的的浮动3D形状
- 每个元素使用相同的设计处理

**布局slop：**
- 忽视内容需求的模板驱动布局
- 所有内容都放在卡片中，无论内容类型
- 过度的空白而没有层次
- 所有元素居中对齐

**文案slop：**
- "Empower your business" 类型的标题
- 无上下文的通用CTA，如 "Get Started"
- 充满流行词的描述
- 股票照片美学

### 质量原则

**内容优先设计：**
- 根据实际内容需求进行设计
- 根据重要性创建层次结构
- 让内容决定布局，而不是模板

**有意的选择：**
- 每个设计决策都应有合理依据
- 使用模式是因为它们服务于用户，而不是因为它们流行
- 根据元素的重要性变化视觉处理

**真实的声音：**
- 文案应反映品牌个性
- 避免通用的营销语言
- 具体说明价值主张

## 脚本

### detect_slop.py

分析文本文件中的AI slop模式。

**用法：**
```bash
python scripts/detect_slop.py <file> [--verbose]
```

**输出：**
- 总体slop评分（0-100）
- 类别特定发现
- 行号和示例
- 可操作的建议

**评分：**
- 0-20：低slop（真实写作）
- 20-40：中等slop（一些模式）
- 40-60：高slop（许多模式）
- 60+：严重slop（高度通用）

### clean_slop.py

自动从文本文件中移除常见的slop模式。

**用法：**
```bash
# 预览更改
python scripts/clean_slop.py <file>

# 保存更改（创建备份）
python scripts/clean_slop.py <file> --save

# 保存到不同文件
python scripts/clean_slop.py <file> --output clean_file.txt

# 激进模式
python scripts/clean_slop.py <file> --save --aggressive
```

**清理内容：**
- 高风险短语
- 冗长的结构
- 元评论
- 过度的模糊
- 流行词
- 冗余限定词
- 空的加强词

**安全性：**
- 覆盖时始终创建 `.backup` 文件
- 预览模式在应用前显示更改
- 保持内容含义（非激进模式）

## 最佳实践

### 预防胜于治疗

**在创建内容时：**
1. 针对特定受众写作
2. 使用具体示例而非抽象
3. 直接切入要点，跳过前言
4. 选择精确的词语，而非给人印象的词语
5. 在考虑内容完成之前进行审查

### 上下文意识清理

并非所有模式都是slop：

**可接受的上下文：**
- 学术写作可能需要更多的模糊
- 法律文件需要特定措辞
- 内部文档可以使用简化
- 技术文档有特定领域的惯例

**始终考虑：**
- 受众是谁？
- 目的是什么？
- 这个模式是否有功能？
- 是否有更好的替代方案？

### 迭代改进

1. **检测** - 运行检测脚本或手动审查
2. **分析** - 理解哪些模式是真正的问题
3. **清理** - 在安全的情况下应用自动清理
4. **审查** - 手动验证更改是否保持含义
5. **精炼** - 手动修复剩余问题

### 质量胜于自动化

脚本是工具，而不是判断的替代品：
- 使用自动检测找到候选项
- 对明显模式应用自动清理
- 手动审查任何改变含义的内容
- 根据上下文行使判断

## 集成模式

### 代码审查

```bash
# 提交前检查文件
python scripts/detect_slop.py src/documentation.md --verbose

# 自动清理
python scripts/clean_slop.py src/documentation.md --save
```

### 内容管道

1. 创建初始内容
2. 运行slop检测
3. 应用自动清理
4. 手动审查和精炼
5. 最终质量检查

### 标准执行

创建项目特定的阈值：
- 最大可接受slop评分：30
- 对于评分 > 20 的内容需要手动审查
- 自动拒绝评分 > 50 的提交

## 限制

**脚本仅处理文本：**
- 代码slop检测为手动（使用 code-patterns.md）
- 设计slop检测为手动（使用 design-patterns.md）

**上下文敏感性：**
- 脚本无法理解所有上下文
- 某些“slop”在特定领域可能是合适的
- 始终审查自动更改

**语言覆盖：**
- 检测模式优化为英语
- 代码模式专注于常见语言（Python, JS, Java）
- 设计模式不依赖于平台

## 常见场景

### 场景 1：审查AI生成的内容

```bash
# 用户询问：“你能审查这篇文章的AI slop吗？”
1. 阅读 references/text-patterns.md 以了解需要注意的模式
2. 运行：python scripts/detect_slop.py article.txt --verbose
3. 审查发现并应用手动清理
4. 可选择运行：python scripts/clean_slop.py article.txt --save
5. 最终手动审查清理后的内容
```

### 场景 2：清理代码库

```bash
# 用户询问：“帮我清理代码中的通用AI模式”
1. 阅读 references/code-patterns.md
2. 手动审查代码文件以查找模式
3. 创建通用名称的重命名列表
4. 按照 code-patterns.md 中的原则重构
5. 移除明显的注释和过度抽象
```

### 场景 3：设计审查

```bash
# 用户询问：“这个设计看起来太通用了吗？”
1. 阅读 references/design-patterns.md
2. 检查高信心slop指标
3. 确定具体问题（渐变、布局、文案）
4. 提供来自 design-patterns.md 的具体建议
5. 建议具体替代方案
```

### 场景 4：建立质量标准

```bash
# 用户询问：“帮我为我们的团队创建质量标准”
1. 审查所有三个参考文件
2. 确定与用户领域最相关的模式
3. 创建项目特定的指南
4. 在开发管道中设置检测脚本
5. 记录可接受的例外情况
```

## 成功的技巧

**对于文本清理：**
- 首先运行检测以了解范围
- 对于重要内容使用非激进模式
- 始终审查自动更改
- 首先关注高风险模式

**对于代码清理：**
- 从重命名通用变量开始
- 接下来移除明显的注释
- 最后重构过度设计的代码
- 在每次重大更改后进行测试

**对于设计清理：**
- 根据模式审核视觉元素
- 优先解决结构性问题而非美学问题
- 确保更改服务于用户需求
- 保持品牌一致性

**一般原则：**
- 质量 > 一致性
- 上下文 > 规则
- 清晰 > 聪明
- 具体性 > 通用性
